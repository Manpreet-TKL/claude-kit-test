# Security & reliability findings (Phase 13)

Estate-wide security and reliability observations for the 102-channel corpus, derived
**deterministically** from the redacted `canonical/` copies. The machine-readable audit
trail is `security/security-scan.csv` (one row per channel, regenerated by
`bin/build_security_scan.py`); every headline number below is a query over that file or
over `canonical/` directly, and each finding names its evidence.

These are findings about the **exported configuration**, not a live-system penetration
test. Nothing here was exercised against a running instance. Severities are engineering
judgement about the configuration as written; several are **conditional on deployment
topology** (network segmentation, firewalling) that the exports do not reveal - those
conditions are the open items in `unresolved/questions.md`, and are called out per
finding rather than assumed worst-case.

Confidence convention: **confirmed** = the setting/value is present in the export;
**inferred** = the operational consequence follows from Mirth's documented semantics or
from the corpus pattern, not from an observed incident.

## Security findings

| ID | Severity | Finding | Scope | Confidence |
|---|---|---|---|---|
| SEC-1 | Medium* | OpenEyes API calls use plain HTTP to the co-located `web` host | 294 HTTP Sender endpoints, 11 of 13 instances | confirmed (transport) / inferred (exposure) |
| SEC-2 | High | A single shared `api` Basic-auth credential is reused estate-wide | 356 occurrences, all instances | confirmed |
| SEC-3 | Medium | Inbound HTTP Listeners accept requests with no authentication | 18 HTTP Listeners, `authType=NONE`, 10 instances | confirmed |
| SEC-4 | Low-Med | NHS Wales MPI has a plaintext endpoint configured alongside the TLS one | 8 Wales query channels, `http://...:23000` | confirmed |
| SEC-5 | Medium | Wales MPI SOAP calls carry no HTTP-Basic / credential layer | 8 Wales query channels | confirmed (config) / trust model unresolved |

| SEC-7 | Medium | 26 channels run in `DEVELOPMENT` message-storage mode | 26 of 102, 7 instances | confirmed (mode) / inferred (retention) |

low, and it rises only if that hop crosses an untrusted network.

### SEC-1 - Plain HTTP to the OpenEyes `web` service
Every OpenEyes PASAPI / Core call in the estate except MEH's targets `http://web/...` -
`web` being the local service name of the co-located OpenEyes web container (e.g.
`http://web/PASAPI/V3/Patient/${HospitalNumber}`, `http://web/api/v1/request/queue/add`).
The endpoint scheme count over connector `<host>` fields is **294 http:// vs 8 https://**;
all 294 point at host `web`, all 8 at `openeyes.moorfields.nhs.uk` (MEH).

- This is a Mirth-to-OpenEyes hop **inside the deployment** (a container/compose service
  name that does not resolve outside it), not plaintext across an NHS or public network.
  On an isolated bridge/loopback segment the exposure is low.
- It nonetheless transmits the shared `api` Basic credential (SEC-2) and full patient
  payloads **unencrypted on that segment**, with no TLS fallback. If the Mirth and OE
  containers are ever not co-located, or the segment is shared with other workloads, the
  credential and PHI are exposed. Whether that segment is isolated is unresolved (network
  ownership, `unresolved/questions.md`).
- **MEH is the counter-example and the pattern to follow:** it has zero `http://web`
  senders and calls `https://openeyes.moorfields.nhs.uk` over TLS - its Mirth reaches OE
  across a real network and correctly encrypts. The estate can do TLS where the topology
  demands it; the plaintext instances simply co-locate and have not needed to.
- **Recommendation:** confirm the Mirth<->OE segment is isolated for every co-located
  instance; where it is not (or to defend in depth), terminate TLS at the OE web tier and
  switch the senders to `https://`. Evidence: `security/security-scan.csv`
  (`outbound_http`/`outbound_https`), `openeyes/api-usage.md`.

### SEC-2 - Single shared service credential, estate-wide
The corpus embeds one 11-char `api` account password, recurring **356 times** identically
across all instances (value withheld; see `secrets/redaction-log.csv` and
`auth/persisted-login.md`). Authentication to OpenEyes is stateless preemptive HTTP Basic
on every call; there is no per-instance or per-channel credential separation.

- **Blast radius:** compromise or rotation of that one credential affects every channel in
  every instance. There is no isolation boundary.
- **Recommendation:** move the credential to a secret store injected at deploy time (the
  model in `deployment/deployment-automation.md` and the `${REDACTED_PASSWORD}` placeholder
  class in `templates/README.md` already assume this), and issue distinct per-instance (or
  per-channel-role) accounts so credentials can be rotated and revoked independently.
  Confidence: confirmed.

### SEC-3 - Unauthenticated inbound HTTP Listeners
All **18 HTTP Listener** source connectors set `authType=NONE` (Bedford, Bolton, EK, ENHT,
MEH x2, Newmedica, Pennine, Portsmouth, Sussex, Wales x8). Any client able to reach the
listener's host:port can submit a request; the channel applies no HTTP-layer authentication
before running.

- Real exposure depends on who can route to the port (SEC-6, network segmentation). The
  channels' own filters may reject malformed input, but that is input validation, not
  authentication.
- **Recommendation:** put these behind an authenticating reverse proxy or enable Mirth
  listener auth (Basic/API-key) where the endpoint is reachable beyond a trusted segment;
  at minimum, document the intended network boundary for each. Evidence: `security-scan.csv`
  (`inbound_auth=NONE`), `networking/port-map.md`.

### SEC-4 - Wales MPI: plaintext endpoint present alongside TLS
Each of the 8 Wales `OpenEyes Query - <board>` channels configures **both**
`https://mpilivequeries.cymru.nhs.uk:443/...` (WSDL + a location URI) **and**
`http://mpilivequeries.cymru.nhs.uk:23000/...` (a plaintext location URI). A patient
demographic query (PDQ) carries identifiers; if the `:23000` plaintext endpoint is the one
actually selected at runtime, the query crosses the network in clear.

- Which endpoint is active cannot be determined from the export alone (Web Service Sender
  endpoint selection). Both are present; the plaintext one is flagged for confirmation.
- **Recommendation:** confirm the `:443` TLS endpoint is the active one and remove or
  firewall the `:23000` plaintext alternative. Evidence: `canonical/Wales/OpenEyes Query -
  *.xml` (`<locationURI>`, `<wsdlUrl>`).

### SEC-5 - Wales MPI SOAP has no credential layer
The Wales MPI Web Service Sender calls carry no HTTP-Basic / WS-Security credential in the
configuration - the trust model appears to be network-level (source IP / private circuit)
rather than message-level auth. This is recorded as unresolved in `auth/persisted-login.md`
and `unresolved/questions.md`.

- **Recommendation:** obtain and document the NHS Wales MPI trust model; if it is
  network-trust only, ensure the network path is genuinely restricted. Confidence: config
  confirmed; trust model unresolved (needs the MPI owner).

### SEC-6 - Listeners bind all interfaces
53 listener-sourced channels bind their source to `0.0.0.0` (all interfaces) rather than a
specific address. Combined with SEC-3, an HTTP listener on `0.0.0.0` with `authType=NONE` is
reachable from any network the host is attached to.

- **Recommendation:** bind listeners to the specific interface that should receive traffic,
  or enforce the boundary at the firewall. Evidence: `security-scan.csv`
  (`listener_binds_all_ifaces`).

### SEC-7 - DEVELOPMENT message-storage mode in production
26 of 102 channels set `messageStorageMode=DEVELOPMENT` (Bedford 5/5, Sussex 5/5, MEH 5/6,
EK 2/7, Optegra 3/6, Portsmouth 3/9, Newmedica 3/35); the other 76 use `PRODUCTION`.
DEVELOPMENT is Mirth's **heaviest-retention** mode: it persists full message content **and
all channel/connector maps** for every message, intended for debugging.

- **Consequence (inferred):** these channels retain complete message bodies - which for the
  PAS/document feeds means **patient data** - on disk for longer and at higher volume than
  PRODUCTION, increasing both the data-retention/PHI-at-rest footprint and disk/IO load. It
  also often signals channels left in a debug setting after go-live.
- **Recommendation:** review each DEVELOPMENT channel; unless active debugging is intended,
  set PRODUCTION (or a lighter mode) and define a content-pruning/retention policy. Evidence:
  `grep messageStorageMode canonical/`.

### SEC-8 - Patient-shaped sample payloads (RESOLVED this project)
Mirth transformer `<inboundTemplate>` / `<outboundTemplate>` fields (UI-only sample messages,
not read at runtime) held patient-shaped sample data (names, DOBs, hospital/NHS numbers) in
166 fields across 75 channels. Wave 0 redaction originally covered only credentials, so these
passed through. **Fixed:** `bin/build_canonical.py` now blanks these fields across
`canonical/` (logged by length only in `secrets/redaction-log.csv`); the template builder
blanks them again defensively. Verified: zero sample identifiers remain in the output tree.
No analytical conclusion depends on these fields. Full detail:
`unresolved/questions.md` and `canonical/NORMALISATION-RULESET.md`.

### Cross-cutting: no secret management
Credentials live inline in the channel exports (redacted to `${REDACTED_PASSWORD}` in our
copies). There is no vault, no environment-variable injection, and no rotation mechanism
evident. SEC-2 is the credential-reuse facet; the structural fix is the deploy-time secret
injection designed in `deployment/deployment-automation.md`, keeping secrets out of version
control entirely (`templates/README.md`, two-placeholder-class model).

## Reliability findings

| ID | Severity | Finding | Scope | Confidence |
|---|---|---|---|---|
| REL-1 | Med-High | Destination queueing is off almost everywhere | 101 of 102 channels | confirmed (config) / inferred (impact) |
| REL-2 | Medium | No send retry on destination failure | 97 of 102 channels `retryCount=0` | confirmed |
| REL-3 | Medium | Newmedica routing depends on runtime globalMap tables | 28-way PAS-In fan-out | inferred, unresolved |
| REL-4 | Medium | Shared-fate: one OE endpoint + one credential per instance | all instances | confirmed |
| REL-5 | Low-Med | Channels process single-threaded | 100 of 102 `processingThreads=1` | confirmed |

### REL-1 - Queueing off; behaviour on downstream failure differs by channel role
Only **1 of 102** destinations enables queueing (`queueEnabled=true`: Newmedica "OpenEyes PAS
Out"). For the other 101, a destination that cannot deliver does **not** durably queue the
message for later; the message errors.

The impact is **not uniform** and must not be over-stated:
- **Synchronous query channels** (Wales PDQ/MPI, patient-search) - queue-off is *correct*:
  the caller is waiting for a live response; a queued reply is useless.
- **Asynchronous data-delivery feeds** (PAS ADT -> PASAPI inbound, document/DICOM ingestion)
  - queue-off means a transient OpenEyes outage yields **errored messages** that need manual
  reprocessing or an upstream resend rather than automatic delivery when OE returns. This is
  the real reliability gap.
- **Recommendation:** enable destination queueing (with a bounded retry/rotation) on the
  asynchronous delivery feeds specifically; leave synchronous query channels as-is. Evidence:
  `security-scan.csv` (`dest_queue_enabled`).

### REL-2 - Minimal retry
97 of 102 channels have every destination at `retryCount=0` - a single delivery attempt, no
retry on transient error. The only channels with retry are Bedford "Document OUT" (3),
Newmedica "DICOM" (3) and "OpenEyes PAS Out" (3), and Portsmouth "General Documents IN" (5)
and "IOL Documents IN" (5) - all file/document/DICOM ingest paths. The PAS inbound feeds,
which carry demographic/appointment updates, retry zero times.

- **Recommendation:** pair a small retry count with REL-1 queueing on the asynchronous feeds
  so a brief OE blip does not drop a message. Evidence: `security-scan.csv` (`dest_retry_max`).

### REL-3 - Newmedica routing single point of failure
Newmedica's 28-way PAS-In fan-out routes on globalMap lookup tables
(`globalMap:hospitalMapping`, `globalMap:channelLookup`, `dataflows/dataflows.md`). If those
maps are not repopulated after a BridgeLink restart (a deploy-script or startup-channel
responsibility), routing fails for all 28 practices at once. Whether they survive restart is
**unresolved** (`unresolved/questions.md`) and should be confirmed before any redeploy.

### REL-4 - Shared fate on endpoint and credential
Within an instance, all channels call the same OE host and authenticate with the same `api`
credential (SEC-2). A single OE endpoint outage or a credential rotation is therefore an
all-channels event for that instance - there is no partial-degradation boundary. The
mitigation is operational (health-checking, staged credential rotation) plus the per-role
credentials recommended in SEC-2.

### REL-5 - Single-threaded processing
100 of 102 channels set `processingThreads=1` (the exceptions: one at 20, one at 5). Each
channel processes messages serially. This bounds throughput and means a slow downstream call
serialises behind it (head-of-line blocking) rather than processing in parallel.

- This is a deliberate default and is often *desirable* for ordered feeds (ADT ordering must
  be preserved), so it is Low-Med, not a defect. Flag it only where throughput is a known
  concern. Evidence: `grep processingThreads canonical/`.

## Priority summary

1. **SEC-2 (High)** - break the single shared credential into per-instance accounts and move
   it to a secret store; this also unblocks safe rotation. Structural fix already designed in
   `deployment/deployment-automation.md`.
2. **REL-1 / REL-2 (Med-High)** - enable queue + bounded retry on the asynchronous PAS and
   document/DICOM delivery feeds (leave synchronous query channels alone).
3. **SEC-1 / SEC-3 / SEC-6 (Med, topology-conditional)** - confirm the Mirth<->OE segment and
   the inbound-listener boundary are isolated; where they are not, add TLS and listener auth.
4. **SEC-7 (Med)** - review the 26 DEVELOPMENT-mode channels for retention/PHI-at-rest.
5. **REL-3 (Med, unresolved)** - confirm Newmedica globalMap restart-persistence before any
   redeploy.
